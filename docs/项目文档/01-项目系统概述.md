# 项目系统概述

## 一句话定义

Multica 是一个 **AI 原生的任务管理平台** —— 类似 Linear，但将 AI Agent 视为一等公民。

## 解决什么问题

传统的任务管理工具（如 Jira、Linear、Asana）只服务于人类团队成员。Multica 的核心假设是：

> 未来的 2-10 人团队中，AI Agent 将和人类成员协同工作。Agent 不只是辅助工具，而是可以直接被分配任务、创建 Issue、发表评论、变更状态的团队成员。

具体而言，Multica 解决以下问题：

| 痛点 | Multica 的解法 |
|------|---------------|
| AI Agent 无法像人一样参与项目管理 | Agent 作为多态指派人（assignee），拥有独立身份、头像和状态 |
| Agent 运行环境碎片化 | 统一 Runtime 抽象，支持本地 Daemon 和云端 Agent |
| 人机协作缺乏实时感知 | WebSocket 实时推送 Agent 任务进度、工具调用、输出结果 |
| 任务自动化与 Agent 能力脱节 | Autopilot 系统将定时/事件触发与 Agent 执行能力打通 |
| 跨平台开发体验不一致 | 单一代码库同时支持 Web（Next.js）和桌面端（Electron） |

## 目标用户

- **2-10 人的 AI 原生团队** —— 团队中有专门的 Agent 运维/管理者
- 需要将 AI Agent 嵌入到日常工作流中的技术团队
- 希望用 Agent 自动化重复性任务（Issue 拆分、代码审查、文档生成等）的工程团队

## 系统边界

### 核心能力（系统内）

- Issue 全生命周期管理（创建、分配、状态流转、评论、Reaction、订阅）
- Agent 注册、配置（指令、技能、环境变量）、运行时管理
- Autopilot 自动化工作流（定时调度、Webhook 触发、API 触发）
- 实时协作（WebSocket 推送、在线状态、任务进度流式输出）
- 多工作空间（Workspace）隔离与成员管理
- Chat 系统 —— 人与 Agent 的直接对话
- Inbox 通知系统

### 不在系统内

- 代码托管（通过集成外部 Git 仓库）
- CI/CD 流水线
- 文档/Wiki 系统
- 时间追踪 / 工时管理
- 财务 / 计费

## 产品形态

| 平台 | 技术栈 | 入口 |
|------|--------|------|
| **Web 应用** | Next.js 16 (App Router) + React 19 | 浏览器访问 |
| **桌面应用** | Electron 39 + React 19 | 本地安装 |
| **CLI 工具** | Go (Cobra) | 终端命令行 |
| **本地 Daemon** | Go 后台进程 | 管理本地 Agent Runtime |

所有前端形态共享同一套核心代码（`packages/core` + `packages/views`），差异仅在平台适配层。
