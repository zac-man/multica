# 防幻觉编码约定

基于项目代码风格，总结出以下 **必须遵守的编码规范** 和 **绝对禁止的反模式**。

## 反模式（绝对禁止）

### 反模式 1：在 Zustand Store 中复制服务端数据

```typescript
// ❌ 绝对禁止 —— 创建两个数据源，必定漂移
const useIssueStore = create((set) => ({
  issues: [],              // 从 API 拿到的数据
  fetchIssues: async () => {
    const data = await api.listIssues();
    set({ issues: data });  // 不要这样做！
  }
}));

// ✅ 正确做法 —— 服务端数据由 React Query 管理
const { data: issues } = useQuery(useIssueListQueryOptions(wsId));
```

**原因**：API 数据属于 React Query Cache，Zustand 只管理客户端状态（筛选、草稿、面板开关）。如果两者都存 Issue 列表，WS 推送更新时只有一个会更新，导致数据不一致。

---

### 反模式 2：在共享包中使用框架特定 API

```typescript
// ❌ 绝对禁止 —— packages/views/ 中不得使用
import { useRouter } from 'next/navigation';
import { useNavigate } from 'react-router-dom';

// ✅ 正确做法 —— 使用平台抽象
import { useNavigation } from '../navigation/context';
const nav = useNavigation();
nav.push('/issues/123');
```

**原因**：`packages/views/` 和 `packages/core/` 被两个应用共享。引入框架 API 会破坏跨平台兼容性。

---

### 反模式 3：Selector 返回不稳定引用

```typescript
// ❌ 绝对禁止 —— 每次渲染都创建新对象 → 无限重渲染
const { filters, viewMode } = useIssueViewStore(s => ({
  filters: s.filters,
  viewMode: s.viewMode,
}));

// ✅ 正确做法 —— 分别选择原始值，或使用 shallow 比较
const filters = useIssueViewStore(s => s.filters);
const viewMode = useIssueViewStore(s => s.viewMode);

// 或
import { useShallow } from 'zustand/shallow';
const { filters, viewMode } = useIssueViewStore(
  useShallow(s => ({ filters: s.filters, viewMode: s.viewMode }))
);
```

**原因**：Zustand 的 Selector 如果每次返回新引用（对象/数组），会导致订阅组件无限重渲染。

---

### 反模式 4：Workspace-scoped Query 不带 wsId 参数

```typescript
// ❌ 绝对禁止 —— 切换工作空间后数据不会自动更新
const { data } = useQuery({
  queryKey: ['issues'],
  queryFn: () => api.listIssues(),
});

// ✅ 正确做法 —— queryKey 包含 wsId
const { data } = useQuery({
  queryKey: ['issues', wsId, filters],
  queryFn: () => api.listIssues({ workspace_id: wsId }),
});
```

**原因**：queryKey 中的 wsId 使得切换工作空间时自动触发新查询，无需手动 invalidat​e。

---

### 反模式 5：WS 事件直接写入 Store

```typescript
// ❌ 绝对禁止 —— 跳过 Cache 层，破坏单一数据源
wsClient.on('issue:updated', (data) => {
  useIssueStore.getState().updateIssue(data);  // 不要这样做！
});

// ✅ 正确做法 —— WS 事件只 invalidat​e Cache
// use-realtime-sync.ts 中的正确模式
wsClient.on('issue:updated', (data) => {
  queryClient.setQueryData(['issue', wsId, data.id], data);
  // 或 invalidat​e 让 React Query 自动 refetch
});
```

**原因**：WS 事件直接写 Store 会绕过 React Query 的一致性保证，导致 Store 和 Cache 之间数据漂移和竞态条件。

---

## 必须遵守的编码规范

### 规范 1：Mutations 默认乐观更新

```typescript
// 所有 Mutation 应该先本地更新，失败时回滚
useMutation({
  mutationFn: (data) => api.updateIssue(id, data),
  onMutate: async (data) => {
    await queryClient.cancelQueries({ queryKey: ['issue', wsId, id] });
    const prev = queryClient.getQueryData(['issue', wsId, id]);
    queryClient.setQueryData(['issue', wsId, id], { ...prev, ...data });
    return { prev };
  },
  onError: (_err, _data, context) => {
    queryClient.setQueryData(['issue', wsId, id], context.prev);
  },
  onSettled: () => {
    queryClient.invalidateQueries({ queryKey: ['issue', wsId, id] });
  },
});
```

**原因**：用户不应该等待服务端响应。乐观更新让界面即时反馈，失败时自动回滚。

### 规范 2：新增共享页面/组件必须放入 packages/

判断流程：

```
1. 这段代码依赖 Next.js API 吗？
   → 是：放在 apps/web/
   → 否：继续

2. 这段代码依赖 react-router-dom 吗？
   → 是：放在 apps/desktop/.../platform/
   → 否：继续

3. 这是纯 UI 组件吗（无业务逻辑）？
   → 是：放在 packages/ui/
   → 否：继续

4. 这是无头逻辑（Store、Hook、API）吗？
   → 是：放在 packages/core/
   → 否：放在 packages/views/
```

### 规范 3：Store 使用工厂 + 单例模式

```typescript
// 1. 工厂函数
export function createAuthStore(options: { storage: StorageAdapter }) {
  return create<AuthState>()(
    persist(
      (set) => ({ /* state and actions */ }),
      { name: 'multica_auth', storage: createJSONStorage(() => options.storage) }
    )
  );
}

// 2. 注册函数
let store: ReturnType<typeof createAuthStore>;
export function registerAuthStore(instance: ReturnType<typeof createAuthStore>) {
  store = instance;
}

// 3. Proxy 单例导出
export const useAuthStore = new Proxy({} as ReturnType<typeof createAuthStore>, {
  get(_, prop) { return (store as any)[prop]; }
});
```

**原因**：解耦 Store 的创建和消费。应用层创建 Store（注入平台适配器），包层只消费。

### 规范 4：CSS 使用设计 Token，禁止硬编码颜色

```css
/* ❌ 禁止 */
.text-red-500 { color: #ef4444; }
.bg-gray-100 { background-color: #f3f4f6; }

/* ✅ 使用语义化 Token */
.text-destructive { color: var(--destructive); }
.bg-background { background-color: var(--background); }
```

### 规范 5：注释只用英文

```typescript
// ❌ 禁止
// 这是一个处理用户登录的函数

// ✅ 正确
// Handle user login flow
```

**原因**：项目面向国际协作，中文注释会给非中文贡献者造成障碍。

### 规范 6：Test 文件跟随被测代码

| 被测代码 | 测试文件位置 |
|---------|-------------|
| `packages/core/auth/store.ts` | `packages/core/auth/store.test.ts` |
| `packages/views/issues/issues-page.tsx` | `packages/views/issues/issues-page.test.tsx` |
| `apps/web/platform/navigation.tsx` | `apps/web/platform/navigation.test.tsx` |

**绝对不要**在 `apps/web/` 测试 `packages/views/` 的组件。如果需要 mock `next/navigation`，说明测试放错位置了。

## 架构速查卡

```
┌─────────────────────────────────────────────────────┐
│                 何时放在哪里                          │
├─────────────────────────────────────────────────────┤
│                                                     │
│  packages/core/     ← Store、Hook、API、Type        │
│    (零 react-dom、零 localStorage、零 process.env)   │
│                                                     │
│  packages/ui/       ← 纯 UI 组件                    │
│    (零 @multica/core 导入)                           │
│                                                     │
│  packages/views/    ← 共享页面/业务组件              │
│    (零 next/*、零 react-router-dom)                  │
│                                                     │
│  apps/web/platform/ ← 唯一的 Next.js API 使用点     │
│  apps/desktop/.../  ← 唯一的 react-router-dom 使用点│
│                                                     │
│  状态规则：                                          │
│    服务端数据 → React Query                          │
│    客户端状态 → Zustand                              │
│    跨切面配置 → React Context                        │
│                                                     │
│  导航规则：                                          │
│    共享代码 → useNavigation() / <AppLink>            │
│    Web 壳   → next/navigation (仅 platform/)        │
│    桌面壳   → react-router-dom (仅 platform/)       │
└─────────────────────────────────────────────────────┘
```
