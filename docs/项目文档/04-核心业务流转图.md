# 核心业务流转图

## 1. Issue 全生命周期

Issue 是系统的核心实体，贯穿了最多的业务逻辑。

```mermaid
flowchart TD
    A[用户/Agent 创建 Issue] --> B[写入 DB + 生成编号<br/>如 MUL-42]
    B --> C[发布 issue:created 事件]
    C --> D{事件监听器分发}
    D --> E[WS Hub → 广播到工作空间]
    D --> F[Subscriber → 自动订阅创建者]
    D --> G[Activity Log → 记录创建活动]
    D --> H[Notification → 通知被指派人]

    I[用户/Agent 更新 Issue] --> J[写入 DB + 检测变更字段]
    J --> K[发布 issue:updated 事件]
    K --> L[WS Hub → 广播]
    L --> M[前端 WS 收到事件]
    M --> N[React Query Cache 更新 / 无效化]
    N --> O[UI 自动刷新]

    P[Agent 被指派到 Issue] --> Q[TaskService 入队]
    Q --> R[Daemon Claim Task]
    R --> S[Agent 执行中...]
    S --> T[流式输出 task:message 事件]
    T --> U[WS Hub → 前端实时渲染]
    S --> V{执行结果}
    V -->|成功| W[task:completed → 自动评论]
    V -->|失败| X[task:failed → 记录错误]
```

### Issue 状态机

```mermaid
stateDiagram-v2
    [*] --> backlog: 创建
    backlog --> todo: 手动变更
    todo --> in_progress: 开始工作
    in_progress --> done: 完成
    in_progress --> cancelled: 取消
    todo --> cancelled: 取消
    backlog --> cancelled: 取消
    done --> in_progress: 重新打开
    cancelled --> todo: 重新打开

    note right of backlog: 未排期
    note right of todo: 已排期待办
    note right of in_progress: 进行中
    note right of done: 已完成
    note right of cancelled: 已取消
```

### Issue 完整数据流（前端视角）

```mermaid
sequenceDiagram
    participant U as 用户
    participant V as Views (packages/views)
    participant C as Core (packages/core)
    participant Q as React Query Cache
    participant A as API Client
    participant S as Go Server
    participant WS as WebSocket Hub

    U->>V: 打开 Issue 列表
    V->>C: useIssueList(wsId, filters)
    C->>Q: Cache Miss → Fetch
    Q->>A: api.listIssues(...)
    A->>S: GET /api/issues?workspace_id=...
    S-->>A: Issue[] + Pagination
    A-->>Q: 写入 Cache
    Q-->>V: 渲染列表

    Note over U,WS: 用户拖拽 Issue 到 In Progress

    U->>V: 拖拽到 in_progress 列
    V->>C: useUpdateIssue()
    C->>Q: Optimistic Update (本地先变)
    Q->>A: api.updateIssue(id, {status: 'in_progress'})
    A->>S: PUT /api/issues/:id
    S->>S: 写 DB + 发布 issue:updated 事件
    S->>WS: Broadcast 到工作空间
    S-->>A: 200 OK
    A-->>Q: Settle (确认/回滚)

    WS-->>C: WS 事件 issue:updated
    C->>Q: 精确 Cache 更新 / Invalidat​e
    Q-->>V: UI 更新（其他用户也看到）
```

## 2. 认证与初始化流程

### Web 端认证（Cookie 模式）

```mermaid
sequenceDiagram
    participant U as 用户浏览器
    participant N as Next.js (Web)
    participant A as API Server
    participant DB as PostgreSQL

    U->>N: 访问 /issues
    N->>N: DashboardGuard 检查
    N-->>U: 重定向到 /login

    U->>N: 输入邮箱 → 点击发送
    N->>A: POST /auth/send-code {email}
    A->>DB: 存储验证码（5分钟过期）
    A-->>N: 200 OK

    U->>N: 输入验证码
    N->>A: POST /auth/verify-code {email, code}
    A->>DB: 验证码校验
    A->>DB: 查找/创建用户
    A->>A: 签发 JWT
    A-->>N: Set-Cookie: HttpOnly + CSRF Token
    N->>N: setLoggedInCookie()
    N-->>U: 重定向到 /issues

    Note over U,DB: 后续请求自动携带 Cookie

    U->>N: /issues 页面加载
    N->>A: AuthInitializer: GET /api/me
    A->>DB: 验证 JWT → 返回用户信息
    A-->>N: User 对象
    N->>A: GET /api/workspaces
    A-->>N: Workspace[] + 成员关系
    N->>N: 水合 WorkspaceStore
    N->>A: WS 连接（Cookie 自动携带）
    A->>A: 验证 Cookie JWT + 工作空间成员
    A-->>N: WS 连接建立 → 开始实时同步
```

### Google OAuth 流程

```mermaid
sequenceDiagram
    participant U as 用户
    participant W as Web (Next.js)
    participant G as Google OAuth
    participant S as API Server

    U->>W: 点击 Google 登录
    W->>G: 重定向到 Google 授权页
    G-->>U: 用户授权
    G-->>W: 回调 /auth/callback?code=xxx

    W->>S: POST /auth/google {code, redirectUri}
    S->>G: 交换 code → Google 用户信息
    S->>S: 查找/创建用户
    S->>S: 签发 JWT
    S-->>W: Set-Cookie (HttpOnly JWT)

    alt Web 端
        W-->>U: 重定向到 /issues 或 /onboarding
    else Desktop 端
        W-->>U: 重定向到 multica://auth/callback?token=jwt
        U->>U: Desktop 通过 Deep Link 接收 Token
    end
```

## 3. Agent 任务执行流程

这是 Multica 最核心的差异化能力 —— Agent 如何从被指派到完成任务的完整链路。

```mermaid
sequenceDiagram
    participant User as 用户
    participant FE as 前端
    participant API as Go Server
    participant Bus as Event Bus
    participant Task as TaskService
    participant Daemon as 本地 Daemon
    participant Agent as AI Agent (Claude/Gemini/...)

    User->>FE: 指派 Issue 给 Agent
    FE->>API: PUT /api/issues/:id {assignee_type: 'agent', assignee_id: ...}
    API->>API: 更新 DB
    API->>Bus: 发布 issue:updated
    Bus->>Task: 触发任务入队
    Task->>API: 创建 agent_task_queue 记录 (status: queued)
    Bus->>Bus: WS 广播 issue:updated + task:dispatch

    Note over Daemon,Agent: Daemon 轮询认领任务

    loop 每 5 秒
        Daemon->>API: GET /api/daemon/runtimes/:id/tasks/pending
        API-->>Daemon: 待处理任务列表
    end

    Daemon->>API: POST /api/daemon/runtimes/:id/tasks/:id/claim
    API->>API: 更新 status: dispatched → running

    Daemon->>Agent: 启动 Agent 执行（读取 instructions + skills）

    loop Agent 执行中
        Agent-->>Daemon: 流式输出 (text / tool_use / tool_result)
        Daemon->>API: POST /api/daemon/tasks/:id/messages {type, content}
        API->>Bus: 发布 task:message
        Bus->>Bus: WS 广播到前端
        Bus-->>FE: 实时渲染 Agent 输出
    end

    Agent-->>Daemon: 执行完毕

    alt 成功
        Daemon->>API: POST /api/daemon/tasks/:id/complete {result}
        API->>API: 更新 status: completed
        API->>API: 自动创建 Agent 评论（如未手动评论）
        API->>Bus: task:completed
    else 失败
        Daemon->>API: POST /api/daemon/tasks/:id/fail {error}
        API->>API: 更新 status: failed
        API->>Bus: task:failed
    end

    Bus->>Bus: WS 广播最终状态
    Bus-->>FE: UI 更新任务状态
```

## 4. 实时同步引擎（前端）

```mermaid
flowchart LR
    subgraph 服务端
        A[DB 写入] --> B[Event Bus]
        B --> C[WS Hub]
    end

    C -->|WebSocket| D[WSClient]

    subgraph 前端 Core
        D --> E[use-realtime-sync.ts]
        E --> F{事件类型}

        F -->|issue:created| G[Invalidat​e Issue List Query]
        F -->|issue:updated| H[精确更新 Cache 中对应 Issue]
        F -->|comment:created| I[Invalidat​e Comment List Query]
        F -->|task:message| J[直接追加到 Task Messages Cache]
        F -->|inbox:new| K[Invalidat​e Inbox + Unread Count]
        F -->|workspace:deleted| L[Toast 通知 + 强制登出]
        F -->|member:removed| M[Toast 通知 + 重定向]

        G & H & I & J & K --> N[React Query Cache 更新]
        N --> O[UI 自动 Re-render]
    end

    E --> P{Reconnect?}
    P -->|Yes| Q[Invalidat​e ALL Query Keys]
    Q --> N
```

## 5. Autopilot 自动化流转

```mermaid
flowchart TD
    A[Cron 调度器<br/>每 30 秒轮询] --> B{有到期的 Trigger?}
    B -->|No| A
    B -->|Yes| C[创建 autopilot_run<br/>status: pending]
    C --> D{并发策略}
    D -->|skip| E{已有活跃 Run?}
    E -->|Yes| F[跳过本次]
    E -->|No| G[继续执行]
    D -->|queue| G
    D -->|replace| H[取消现有 Run]
    H --> G

    G --> I{执行模式}
    I -->|create_issue| J[创建 Issue<br/>指派给 Agent]
    J --> K[入队 Agent Task]
    I -->|run_only| K

    K --> L[Agent 执行]
    L --> M{Task 完成}
    M --> N[更新 run status: done]
    M --> O[Advance next_run_at<br/>按 Cron 表达式]
```

## 6. 多租户隔离模型

```mermaid
flowchart TB
    subgraph 请求进入
        A[HTTP Request] --> B[Auth Middleware<br/>验证 JWT/PAT]
        B --> C[Workspace Middleware<br/>读取 X-Workspace-ID]
        C --> D{Membership Check<br/>DB 查询 member 表}
        D -->|Not Member| E[403 Forbidden]
        D -->|Is Member| F[注入 member 到 Context]
    end

    subgraph 数据隔离
        F --> G[Handler 执行]
        G --> H[sqlc Query<br/>WHERE workspace_id = $N]
        H --> I[仅返回该工作空间数据]
    end

    subgraph WS 隔离
        J[WS 连接] --> K[验证 membership]
        K --> L[加入 workspace Room]
        L --> M[仅接收该 Room 的广播]
    end

    subgraph 存储隔离
        N[Zustand Store] --> O[createWorkspaceAwareStorage]
        O --> P[Key: multica_xxx:{wsId}]
        P --> Q[切换空间时自动 Rehydrate]
    end
```
